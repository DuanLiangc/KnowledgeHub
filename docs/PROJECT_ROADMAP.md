# Knowledge Hub 项目主线

> 本文档是 Knowledge Hub 后续开发的范围基准。每次开发优先完成当前阶段的验收标准，再进入下一阶段。

- 目标读者：正在从 Java 后端转向 AI 应用开发的初学者
- 项目形态：Spring Boot 模块化单体企业知识库
- 主框架：Spring AI
- 数据与向量存储：PostgreSQL + pgvector
- 更新日期：2026-07-11

## 1. 项目目标

通过一个持续演进的企业知识库项目，学习并实践以下能力：

1. 调用大模型并理解 Token、上下文、流式输出和结构化输出。
2. 处理企业文档，完成解析、切分、Embedding 和向量检索。
3. 构建带引用、能拒答、可评估的 RAG 问答系统。
4. 实现知识空间隔离、文档权限和 Prompt Injection 防护。
5. 建立日志、成本、延迟、限流、降级和部署能力。
6. 在主流程稳定后，再学习受控的 Tool Calling、Agent 和 MCP。

最终交付物不是多个互不相关的 Demo，而是一个可以演示、测试、部署和讲解的完整项目。

## 2. 开发原则

### 2.1 始终围绕一条项目主线

所有学习内容都优先加入 Knowledge Hub。例如学习 Tool Calling 时，工具应查询文档状态或重新触发索引，而不是另做订单系统。

### 2.2 一次只引入一个主要概念

先完成普通数据库持久化，再处理文档；先完成检索，再调用大模型生成答案；先做好单 Agent，再考虑多 Agent。这样出现问题时更容易定位原因。

### 2.3 每个阶段必须可验证

“看完教程”和“代码可以编译”不算完成。每个阶段至少需要：

- 一个可以运行的业务闭环；
- 对关键行为的自动化测试；
- 可以复现的操作说明；
- 对本阶段核心概念的学习记录；
- 通过本阶段列出的验收标准。

### 2.4 优先理解，再使用框架封装

使用 Spring AI 提高开发效率，但仍需理解模型请求、Embedding、Chunk、TopK、Rerank 和上下文组装实际做了什么。

### 2.5 安全和权限不是收尾功能

企业知识库必须在检索时过滤无权访问的数据。不能先检索全部内容，再在答案生成后补权限检查。

## 3. 主流程

```text
上传文档
  -> 保存原始文件和文档记录
  -> 解析文本
  -> 按规则切分为 Chunk
  -> 调用 Embedding 模型生成向量
  -> 写入 pgvector
  -> 根据用户问题召回相关 Chunk
  -> 权限过滤与 Rerank
  -> 组装 Prompt 上下文
  -> 大模型生成答案
  -> 流式返回答案和引用来源
  -> 记录评估、反馈、延迟和 Token 成本
```

## 5. 分阶段实施计划

### 阶段 0：项目骨架

**学习重点**

- Maven 多模块项目。
- 模块化单体与微服务的区别。
- 领域层、应用层、基础设施层和 API 层的职责。
- 端口与适配器：应用层定义接口，基础设施层提供实现。

**已有交付物**

- [x] `knowledge-domain`、`knowledge-application`、`knowledge-infrastructure`、`knowledge-api` 四个模块。
- [x] 创建和查询知识空间 API。
- [x] 内存 Repository 实现。
- [x] 参数校验、统一异常响应和健康检查。
- [x] Controller 测试和 Spring 上下文测试。

**完成标准**

- [x] `mvn clean verify` 执行成功。
- [x] 可以通过 HTTP 创建和查询知识空间。
- [x] 能够按照 [项目结构说明](PROJECT_STRUCTURE.md) 讲清一次请求如何流过各模块。

### 阶段 1：PostgreSQL 与数据基础

**学习重点**

- 数据库迁移工具 Flyway 为什么比手工建表可靠。
- JPA Entity 与领域对象为什么分开。
- Repository Port 如何替换内存 Adapter。
- 数据库配置、环境变量和敏感配置管理。

**项目任务**

- 使用 Docker Compose 启动 PostgreSQL。
- 使用 Flyway 创建知识空间表。
- 新增 PostgreSQL Repository Adapter，替换内存实现。
- 增加按 ID 查询、修改和删除知识空间。
- 使用 Testcontainers 验证真实 PostgreSQL 行为。

**验收标准**

- [ ] 应用重启后知识空间数据仍然存在。
- [ ] 空数据库启动时 Flyway 能自动建表。
- [ ] 数据库连接信息来自环境变量，不提交真实密码。
- [ ] CRUD API 和数据库集成测试全部通过。
- [ ] 能解释领域对象、JPA Entity 和 API DTO 各自的职责。

### 阶段 2：LLM 与模型调用基础

**学习重点**

- System、User、Assistant Message。
- Token、上下文窗口、Temperature、Top-P 和 Max Tokens。
- 普通响应与流式响应的区别。
- 超时、错误响应、Token 用量和模型成本。

**项目任务**

- 在应用层定义 `ChatModelPort`，隔离具体模型供应商。
- 通过环境变量配置 API Key、Base URL 和模型名称。
- 实现一个最小问答 API。
- 使用 SSE 返回流式内容，并支持客户端中断。
- 记录模型名称、耗时和 Token 用量，不记录 API Key。
- 为测试提供 Fake Model，避免自动化测试消耗真实 Token。

**验收标准**

- [ ] 未配置 API Key 时给出清晰错误，不影响非 AI 接口启动和测试。
- [ ] 普通问答和 SSE 流式问答都可调用。
- [ ] 自动化测试不访问真实模型 API。
- [ ] 能解释一次模型请求的消息、参数、响应和费用来源。

### 阶段 3：Prompt 工程与 Spring AI

**学习重点**

- 角色、任务、输入约束、输出格式和 Few-shot。
- JSON Schema、结构化输出校验与失败重试。
- Spring AI 的 `ChatClient`、Prompt Template、模型抽象和 Chat Memory。
- 短期会话记忆与完整历史消息的成本差异。

**项目任务**

- 正式接入 Spring AI，并保留应用层模型端口。
- 把 Prompt 从业务代码中提取为可参数化模板。
- 实现“问题分析”结构化输出。
- 增加会话 ID、多轮消息和历史裁剪。
- 为 Prompt 增加名称和版本字段。

**验收标准**

- [ ] 结构化输出能通过 Java 类型或 Schema 校验。
- [ ] 格式错误时有限重试，不发生无限循环。
- [ ] 不同会话的历史消息互相隔离。
- [ ] Prompt 变更可追踪，并有固定测试样例。

### 阶段 4：文档管理

**学习重点**

- 文件元数据与文件内容的区别。
- 同步上传与异步处理。
- 内容哈希、幂等和重复上传检测。
- 文档处理状态机和失败恢复。

**项目任务**

- 实现 Markdown 和 PDF 上传。
- 保存原始文件、文件名、类型、大小和 SHA-256 内容哈希。
- 建立 `UPLOADED`、`PARSING`、`INDEXING`、`READY`、`FAILED` 状态。
- 提供文档列表、详情、删除和失败重试 API。
- 第一版使用本地文件系统，并通过存储端口隔离实现。

**验收标准**

- [ ] 不允许上传不支持的格式或超过限制的文件。
- [ ] 相同内容重复上传时有明确处理策略。
- [ ] 用户能查看处理状态和失败原因。
- [ ] 删除文档时数据库记录与原始文件保持一致。
- [ ] 文件存储与业务代码之间通过接口隔离。

### 阶段 5：RAG v1

**学习重点**

- RAG 为什么能补充模型未知或私有知识。
- Embedding、向量维度和余弦相似度。
- Chunk Size、Chunk Overlap、TopK 和相似度阈值。
- pgvector 的向量存储与检索。
- 上下文组装、引用来源和拒答 Prompt。

**项目任务**

- 解析 Markdown 和 PDF 文本。
- 先实现固定长度或按段落切分，并记录 Chunk 顺序和来源页码。
- 批量调用 Embedding 模型。
- 使用 pgvector 保存和检索 Chunk 向量。
- 实现“问题 -> 检索 -> 上下文 -> 模型 -> 答案与引用”闭环。

**验收标准**

- [ ] 至少导入 10 份测试文档并完成索引。
- [ ] 可以查看每份文档产生的 Chunk 数量。
- [ ] 答案返回文档 ID、文档名和 Chunk 来源。
- [ ] 没有可靠召回结果时明确回答不知道。
- [ ] 删除或更新文档后，旧 Chunk 和向量不会残留。
- [ ] 核心 RAG 流程有不依赖真实模型的自动化测试。

### 阶段 6：检索评估

**学习重点**

- 召回率、精确率、Recall@K、答案准确率和引用命中率。
- 为什么不能仅凭主观感觉调整 RAG 参数。
- Golden Dataset 的问题、标准答案和相关文档设计。

**项目任务**

- 建立首批 20 至 30 条人工评估数据。
- 编写离线评估任务，保存每次运行的模型、Prompt 和检索参数。
- 对比不同 Chunk Size、Overlap、TopK 和阈值。
- 输出成功样例、失败样例、指标和耗时报告。

**验收标准**

- [ ] 评估命令可重复执行，结果可保存和对比。
- [ ] 每条问题至少标记一个期望命中的来源文档。
- [ ] 能用数据说明当前 Chunk 和 TopK 的选择依据。
- [ ] 新 Prompt 或新模型上线前可以运行回归评估。

### 阶段 7：RAG 进阶

**学习重点**

- BM25 关键词检索与向量语义检索的互补关系。
- 混合检索、多路召回、结果融合和 Rerank。
- Query Rewrite 的价值与风险。
- HNSW 的基本原理，以及近似检索的性能取舍。

**项目任务**

- 增加 PostgreSQL 全文检索或独立 BM25 实现。
- 合并关键词和向量召回结果。
- 接入 Rerank 模型，对候选 Chunk 重新排序。
- 加强低置信度拒答和引用一致性校验。
- 使用阶段 6 的同一评估集比较三种检索方案。

**验收标准**

- [ ] 报告向量、混合、混合加 Rerank 三组指标。
- [ ] 同时记录准确率、P95 延迟和单次成本。
- [ ] 可以通过配置开关启停 Query Rewrite 与 Rerank。
- [ ] 参数调整有评估数据支撑。

### 阶段 8：权限与 AI 安全

**学习重点**

- 用户、角色、知识空间成员和文档 ACL。
- 多租户隔离与普通 RBAC 的区别。
- Prompt Injection、文档内容注入和间接注入。
- 敏感数据脱敏、外部模型数据边界和审计。

**项目任务**

- 引入用户身份和知识空间成员关系。
- 在检索查询阶段加入权限过滤。
- 增加越权测试，证明用户无法召回无权限 Chunk。
- 区分系统指令、用户问题和检索文档内容。
- 对上传、删除、检索和模型调用记录审计日志。

**验收标准**

- [ ] 两个不同权限用户的检索结果符合预期。
- [ ] 无权限 Chunk 不会进入发送给大模型的上下文。
- [ ] 恶意文档中的指令不能覆盖系统规则。
- [ ] 日志不包含 API Key、密码或不必要的文档全文。
- [ ] 高风险操作有二次确认或明确拒绝策略。

### 阶段 9：生产工程化

**学习重点**

- 模型调用的超时、重试、限流、熔断和降级。
- Token、延迟、错误率、召回数量和用户反馈指标。
- 缓存的适用范围与数据过期风险。
- 异步任务、失败重试、幂等和链路追踪。
- Java 21 虚拟线程适合阻塞 I/O 的原因，以及它与 Reactor 的取舍。

**项目任务**

- 接入 Micrometer 和 OpenTelemetry。
- 为模型调用、Embedding、检索和 Rerank 增加 Trace。
- 统计请求 Token、费用、耗时和错误原因。
- 增加限流、有限重试和备用模型配置。
- 为文档解析与索引增加可靠异步执行和失败恢复。
- 对模型调用、批量 Embedding 或文档处理做并发实验，比较平台线程、虚拟线程和 Reactor 的适用场景。

**验收标准**

- [ ] 能看到一次问答各阶段的耗时。
- [ ] 模型超时、限流和不可用时返回可理解的错误。
- [ ] 重试不会导致重复文档或重复 Chunk。
- [ ] 可以按模型、用户和日期统计 Token 与成本。
- [ ] 用户可以提交有用或无用反馈。
- [ ] 有压测数据说明是否启用虚拟线程，不在同一链路中无依据地混用虚拟线程与 Reactor。

### 阶段 10：受控 Tool Calling

**学习重点**

- Function Schema、参数描述、参数校验和工具结果回传。
- 只读工具与有副作用工具的风险差异。
- 权限校验、幂等、白名单、确认和审计。

**项目任务**

- 提供查询文档处理状态工具。
- 提供查询知识空间统计工具。
- 可选提供“重新触发失败索引”工具，并要求权限与确认。
- 展示工具名称、参数、结果、耗时和失败原因。

**验收标准**

- [ ] 模型不能调用未注册工具。
- [ ] 所有参数都经过服务端校验，不能相信模型参数。
- [ ] 工具内部再次执行用户权限校验。
- [ ] 有副作用的工具需要确认并写入审计日志。
- [ ] 工具超时或失败不会导致无限调用循环。

### 阶段 11：最小 Agent、MCP 与部署

**学习重点**

- Agent Loop、ReAct 和单步/多步工具调用。
- Agent 最大步数、终止条件和预算限制。
- MCP Client、Server、Tool、Resource 和 Prompt 的职责。
- Docker 镜像、环境配置、健康检查和失败恢复。

**项目任务**

- 实现一个只围绕知识库工作的单 Agent。
- 限制 Agent 可用工具、最大执行步数和 Token 预算。
- 接入一个现成 MCP Server，跑通工具发现与调用。
- 可选将一个只读知识库能力暴露为 MCP Server。
- 使用 Docker Compose 部署应用、PostgreSQL 和必要依赖。

**验收标准**

- [ ] Agent 能完成一个需要两步工具调用的明确任务。
- [ ] Agent 每一步都有可读执行日志并能正常终止。
- [ ] MCP 调用遵守与普通 Tool Calling 相同的权限规则。
- [ ] 新环境可以按文档启动完整系统。
- [ ] 服务重启后数据和索引保持一致。

多 Agent、Planner-Worker-Critic 和长期记忆不属于本阶段必做内容。只有单 Agent 的评估、安全和可观测性稳定后，才进入这些扩展主题。

### 阶段 12：项目包装与面试准备

**学习重点**

- 如何用业务问题、技术取舍和量化结果表达项目。
- 如何解释 RAG、Chunk、Rerank、权限过滤和 Agent 风险。
- 如何展示 Java 并发、Reactor、JVM 和工程化经验的迁移价值。

**项目任务**

- 完善项目 README、部署文档、API 文档和架构图。
- 整理评估指标、延迟、成本及优化前后对比。
- 录制完整演示：上传、索引、问答、引用、权限和失败恢复。
- 准备项目讲稿和常见面试问题。

**验收标准**

- [ ] 新开发者能按照 README 独立启动项目。
- [ ] 项目演示不依赖手工修改数据库。
- [ ] 架构图与实际代码和部署保持一致。
- [ ] 项目结论包含真实评估数据，不虚构线上用户量或收益。
- [ ] 能在 5 分钟内讲清背景、架构、难点、取舍和结果。

## 6. 暂不纳入主线

以下内容有价值，但不应阻塞企业知识库第一版：

- 同时深入 Spring AI 和 LangChain4j；
- Milvus、Pinecone、Weaviate 等额外向量数据库；
- 多 Agent 和复杂 Planner-Worker-Critic 编排；
- 跨会话长期记忆；
- Kubernetes 和大规模微服务拆分；
- 模型微调和训练；
- 语音、视频等复杂多模态能力；
- 独立订单、库存或客服 Demo。

当主线全部完成、评估数据证明需要，或者目标岗位明确要求时，再建立独立扩展计划。

## 7. 阶段推进规则

每次开始新阶段前执行下面的检查：

1. 上一阶段的验收项已经完成，未完成项有明确原因。
2. `mvn clean verify` 通过。
3. README、配置示例和架构文档与代码一致。
4. 没有提交 API Key、数据库密码或用户隐私数据。
5. 本阶段新增的重要概念已经能用自己的话解释。
6. 本文档的状态列与实际进度同步更新。

如果路线与实际开发冲突，先修改本文档并记录原因，再扩大项目范围。
